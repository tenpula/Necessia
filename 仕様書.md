# **要件定義書: Research Gap Visualizer (CS Edition)**

## **1\. プロジェクト概要**

### **1.1 プロダクトの定義**

情報工学（Computer Science）分野の研究者を対象とし、論文間の引用文脈（支持・批判・手法・比較）を可視化し、論理的な分析に基づいて「研究の空白（Gap）」や「未引用の関連研究」を指摘するWebサービス。

### **1.2 コアコンセプト**

* **Logic over Creativity:** 突飛な発想ではなく、堅実な論理と類似性に基づく「見落とし」の発見。  
* **CS-Centric:** arXivやカンファレンス論文（Conference Proceedings）を中心としたエコシステムへの対応。  
* **Cost-Efficiency:** キャッシュと集合知を活用し、個人開発でも維持可能な低コスト運用。

### **1.3 ターゲットユーザー**

* 情報工学分野の修士・博士課程学生、研究者。  
* 自分の研究がSOTA（State-of-the-Art）と比較して正しく位置づけられているか確認したい人。

## **2\. 機能要件 (Functional Requirements)**

### **2.1 入力インターフェース**

英語論文のみを対象とする。

1. **Seed Paper Upload:** 自身のドラフト（英語PDF）のアップロード。  
2. **Identifier Search:** arXiv URL、DOI、または論文タイトルでの検索。  
   * *特化機能:* CS分野で頻繁に使われる **arXiv URL** の直接入力に対応。

### **2.2 ネットワーク可視化（メインビュー）**

ノードとエッジによるグラフ描画。

* **ノード表現:**  
  * サイズ：被引用数（Citation Count）に基づく。  
  * 色分け：出版年（古い→新しい）。  
  * 形状：Journal Paper（円）と Conference Paper（四角）の区別（CS分野では会議録が重要なため）。  
* エッジ表現（Semantic Edges） \- CS特化チューニング:  
  LLMにより引用文脈（Citation Context）を解析し、以下の4種に分類。  
  * 🟢 **手法利用 (Methodology):** "We use the method proposed by..." (手法の借用)  
  * 🔴 **批判・制限 (Critique/Limitation):** "However, their approach fails to..." (欠点の指摘)  
  * 🟣 **比較 (Comparison):** "We compare our results with..." (ベースラインとしての比較)  
  * ⚪ **背景 (Background):** 一般的な関連研究としての言及。

### **2.3 アイデア・スパーク機能（The "Gap" Finder \- Logic Driven）**

「論理的な必然性」があるにもかかわらず、接続されていない関係を提示。

* **Missing Link / Gap Proposal:**  
  * ベクトル類似度が高く、かつ「共通の引用元（Co-citation）」が多いペアを検出。  
  * **提示ロジック例:** 「論文Aと論文Bは共に論文C（Transformer）を引用しており、扱うタスクも同じですが、お互いを引用していません。比較実験の価値があります。」

### **2.4 分析・詳細パネル**

* **Context Snippet:** エッジをクリックすると、実際の論文内の「引用されている文章」を表示（ハルシネーション対策）。  
* **Submission Check:** ユーザー論文に対し、「このトピックの論文は通常、論文Xと論文Yを引用しますが、あなたの論文にはありません」と警告。

## **3\. 技術スタック・低コスト戦略 (Technical Constraints)**

### **3.1 データソース**

* **OpenAlex API (Main):** 無料枠（10万リクエスト/日）が非常に大きく、個人開発に最適。  
  * *CS対応:* arXiv IDでの検索に対応しているため親和性が高い。  
* **Semantic Scholar API (Partner):** 補完的に利用するが、レート制限が厳しいため、OpenAlexで見つからない場合のみのフォールバックとする。

### **3.2 AI/LLM処理とコスト削減**

* **モデル選定:**  
  * Gemini 1.5 Flash。単価が安く、分類タスクにおいて十分な性能を持つモデルを選定。  
  * Temperature: 0.0 〜 0.2 に設定し、創造性よりも分類の一貫性を重視。  
* **「集合知」キャッシュシステム (重要):**  
  * 一度解析した論文ペアの「引用文脈（支持/批判etc）」は、Firestore等のDBに永続保存する。  
  * ユーザーAが論文Xの引用関係を解析したら、ユーザーBが論文Xを見たときはAPIを呼ばず、DBの結果を即座に返す。  
  * これにより、ユーザーが増えるほどAPIコストが下がり、表示速度が上がる。

### **3.3 インフラ・開発**

* **Frontend:** React (Vite) \+ React Flow (グラフ描画)。  
* **Backend:** Firebase Functions (サーバーレス) または Next.js API Routes (Vercel)。  
* **Database:** Firestore (キャッシュデータ、ユーザーデータ保存)。  
* **PDF Parser:** pdf.js (クライアントサイド処理推奨)。サーバー負荷を減らすため、テキスト抽出は可能な限りブラウザ側で行う。

## **4\. UI/UX 要件**

* **非同期処理の明示:**  
  * 処理速度が遅くなっても良いという要件に基づき、解析中は「Analyzing citation contexts... (3/20)」のようなプログレスバーを表示し、完了した部分から順次グラフを描画する（ストリーミング的なUX）。  
* **英語UI:** ターゲット層とデータ言語に合わせてUIも英語で統一。

## **5\. 開発フェーズ (Phases)**

### **Phase 1: MVP (検索・表示のみ)**

* arXiv URLを入力 → OpenAlexから引用リスト取得 → シンプルなグラフ表示。  
* LLMによる文脈解析は**行わず**、まずは構造のみ可視化。

### **Phase 2: Context & Caching (意味付け)**

* LLM導入。引用文脈の分類（支持/批判/比較/背景）を実装。  
* Firestoreへのキャッシュ機構の実装。  
* この段階でコストと精度のバランスを見る。

### **Phase 3: Gap Finding & Upload (提案機能)**

* PDFアップロード機能の実装。  
* ベクトル検索による「Gap」提案機能の実装。
