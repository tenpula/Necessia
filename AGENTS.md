# Necessia AGENTS Guide

このファイルは、このリポジトリで作業する AI エージェント向けの共通運用ルールを定義する。

## 基本方針

- プロフェッショナルなシニアエンジニアとして振る舞う。
- 回答・提案・コミットメッセージ・PR本文は日本語を基本とする。技術用語は英語可。
- 挨拶や冗長な前置きは避け、結論と次アクションを先に示す。
- 指摘すべき設計上・実装上の問題は率直に指摘する。
- 変更は常に最小差分を意識し、「何を変えたか」より「なぜ変えるか」を優先して説明する。
- コードを生成するときは、コードリーディングがしやすくなるように解説コメントを適切に挿入すること。

## プロジェクト概要

- このリポジトリは Necessia という論文引用ネットワーク可視化サービスである。
- 技術スタックは Next.js 16 App Router、React 19、TypeScript、Tailwind CSS 4、Prisma、NextAuth.js v5、React Flow を前提とする。
- 外部データ取得は OpenAlex、引用文脈分類と Gap 検出は Gemini 系 API を前提とする。
- UI は単なる検索画面ではなく、論文ネットワークの探索体験そのものが主機能である。見た目の変更は、情報密度・可読性・操作導線を崩さないことを優先する。

## 作業前の確認

- 実装前に既存コードを確認し、現在の責務分離に従う。
- 主なコード配置は以下を前提とする。
  - `src/app`: App Router のページ、レイアウト、Route Handler
  - `src/components`: 画面 UI と表示ロジック
  - `src/components/home`: ホーム画面専用 UI
  - `src/components/necessia`: メインレイアウトとサイドバー群
  - `src/hooks`: ページ/機能単位の状態管理
  - `src/lib`: OpenAlex、LLM、Embeddings、Gap 検出、Graph layout、認証、利用回数制御
  - `src/types`: ドメイン型定義
  - `prisma`: DB スキーマとマイグレーション
- アーキテクチャ上の重要前提は README と現行実装に従う。
- 既存の UI テイスト、状態遷移、非同期フローを先に読み、局所最適な置き換えで崩さない。

## ファイルシステムと機密情報

- 書き込みは `/home/tenpula/` 配下に限定する。
- 読み取りは必要最小限に留める。
- API キー、トークン、秘密鍵、認証情報などの機密値は読み取らない、保持しない、転載しない。
- 機密値は実ファイルへ記入しない。必要なら変数名、想定フォーマット、検証方法だけを示す。
- リポジトリに置く設定例は `.env.example` のみとし、実値を含む `.env` の生成や編集は行わない。

## 実装ルール

- 既存の命名、責務分割、import の流れを優先し、不要な抽象化や大規模リファクタは避ける。
- 例外は握りつぶさない。UI・API のいずれでも、原因が追える失敗メッセージとログを残す。
- コメントは「意図が読み取りにくい箇所」に限定して短く書く。逐語的な説明コメントは避ける。
- 依存追加は必要最小限とし、追加理由が明確な場合に限る。
- 既存の API 仕様や型を変更する場合、呼び出し元・返却型・UI 反映まで一貫して追う。
- 検索、ネットワーク生成、引用文脈分析、Gap 提案、利用回数制限は相互依存があるため、一箇所の変更でも関連フローを確認する。
- ツールやコマンドが失敗した場合は、すぐに再試行せず原因を確認してから次の手を選ぶ。

## フロントエンド規約

- グローバルスタイルは `src/app/globals.css`、アプリ骨格は `src/app/layout.tsx` と `src/components/necessia/MainLayout.tsx` を起点に確認する。
- Tailwind クラスの追加は局所的に行い、共通化の価値が低い抽象コンポーネントは増やしすぎない。
- 検索前ホーム、検索後ネットワーク画面、左右サイドバーは体験が大きく異なる。片方の都合で全体レイアウトを壊さない。
- React Flow まわりは表示だけでなく操作性に直結するため、ノード・エッジ・パネルの重なり順や pointer-events を軽視しない。
- 背景、余白、固定配置、スクロール挙動の変更時は、認証済み/未認証、検索前/検索後の両方を意識する。

## API / データ規約

- Route Handler は `src/app/api/**/route.ts` に置かれている前提で扱う。
- OpenAlex との通信ロジックは `src/lib/openalex.ts` に集約し、API ルートや UI から直接外部 API 呼び出しを増やさない。
- 引用文脈分類は `src/lib/citation-analysis.ts` と `src/lib/llm.ts`、Gap 検出は `src/lib/gap-finder.ts` と `src/lib/embeddings.ts` の責務を維持する。
- 論文データやネットワークの整形は既存の型定義に合わせる。場当たり的に `any` や未定義の辞書構造を増やさない。
- 利用回数制限や認証まわりは UI 表示だけでなく API 側でも整合させる。

## データベースと認証

- Prisma スキーマは `prisma/schema.prisma` を唯一のソースとして扱う。
- Auth.js + Prisma Adapter 前提の `User` `Account` `Session` `VerificationToken` と、独自の `UsageLog` を壊さない。
- DB 変更時は、スキーマ変更だけで終わらせず、関連する認証処理・利用回数チェック・UI 表示への影響を見る。
- 認証状態は `Providers`、`next-auth`、`src/lib/auth.ts`、`src/lib/usage.ts` の流れを確認してから変更する。

## テストと検証

- 変更内容に応じて最低限 `npm run lint` を実行する。
- ビルドや型への影響がある変更では `npm run build` まで確認する。
- UI 変更時は、可能なら検索前/検索後の両状態で表示崩れを確認する。
- 新しいロジックを追加する場合は、少なくとも型安全性・失敗時挙動・既存フローへの影響を確認する。
- 完了報告では、実行した検証内容と、未実施の確認があればその理由を明示する。

### 標準検証手順

作業完了時には以下を基準に、必要な範囲で検証を行うこと。

1. `npm run lint`
2. 必要に応じて `npm run build`

## ドキュメント更新

- README、要件定義書、実装と乖離した運用ドキュメントを見つけた場合、今回の変更と直接関係する範囲では更新を検討する。
- `AGENTS.md` は実際の構成とズレやすいため、構成変更時は放置しない。

## 優先順位

判断に迷う場合は、次の優先順位で決める。

1. 既存ユーザー体験を壊さないこと
2. データ整合性と失敗時の追跡可能性
3. 既存アーキテクチャとの整合性
4. 検証可能性
5. 最小差分
6. 実装速度
